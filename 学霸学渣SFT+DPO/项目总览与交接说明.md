# 学霸学渣 SFT + DPO 项目总览与交接说明

## 1. 项目是什么

这是一个基于 `Qwen/Qwen2.5-3B-Instruct` 的后训练实验项目，目标不是单纯提升通用问答分数，而是研究一种“按题目难度自适应控制回答长度”的行为：

- `simple` 题尽量直接短答
- `complex` 题允许或倾向输出较长推理

项目当前已经完成了从环境准备、数据读取、SFT 训练、DPO 训练到统一评测的一整套最小可运行闭环。

## 2. 项目已经实现了什么

当前已经完成的工作如下：

- 本地拉取并验证了基础模型 `Qwen/Qwen2.5-3B-Instruct`
- 搭建了可运行的训练环境：
  - `transformers`
  - `trl`
  - `peft`
  - `datasets`
  - `accelerate`
  - `bitsandbytes`
  - `torch`
- 完成了 Qwen2.5 的 smoke test，确认：
  - 模型可加载
  - tokenizer 可加载
  - 前向可运行
  - 生成可运行
- 基于同一份数据构造了两条 SFT 路线：
  - `SFT-CoT`：默认倾向长推理
  - `SFT-NoCoT`：默认倾向短答
- 基于 `SFT-CoT best checkpoint` 又训练了一版 `DPO-Adaptive`
- 对以下模型做了统一评测：
  - `Base direct`
  - `Base cot`
  - `SFT-CoT`
  - `SFT-NoCoT`
  - `DPO-Adaptive`

## 3. 项目核心思路

这套实验的核心不是“学霸/学渣”人格本身，而是“是否应该展开推理”这件事。

更准确地说：

- `SFT-CoT` 让模型习惯输出 `<think> ... </think> + <answer> ... </answer>`
- `SFT-NoCoT` 让模型习惯直接给短答案
- `DPO-Adaptive` 用 `difficulty` 条件构造偏好：
  - `simple`：偏好短答
  - `complex`：偏好长推理

也就是说，这个项目本质上是在做一个“推理深度自适应”的对齐实验。

## 4. 数据集说明

数据位于：

- `data/dataset_release/final_clean_527/`

当前最终保留样本数：

- 总计 `527`
- `train = 367`
- `val = 52`
- `test = 108`

关键文件：

- `master_clean_dataset_final.jsonl`
  - 主表，每题同时包含 `question / gold_answer / cot_response / nocot_response / difficulty / question_type / subset / metadata`
- `sft_cot_train.jsonl`
- `sft_cot_val.jsonl`
- `sft_nocot_train.jsonl`
- `sft_nocot_val.jsonl`
- `dpo_adaptive_train.jsonl`
- `dpo_adaptive_val.jsonl`
- `test_eval.jsonl`

DPO 偏好构造规则：

- `simple`：`chosen = nocot_response`，`rejected = cot_response`
- `complex`：`chosen = cot_response`，`rejected = nocot_response`

这份数据的设计重点是：

- 让简单题更偏向直接回答
- 让复杂题更偏向推理回答

## 11. 当前可以怎么解读这些结果

### 最强整体模型

如果只看整体准确率：

- 当前表现最好的是 `SFT-NoCoT`
- `overall accuracy = 0.2130`

它比 `Base direct` 更好，也明显比 `SFT-CoT` 和 `DPO-Adaptive` 更稳。

### 最明显“复杂题更长”的模型

如果只看 `length gap = complex avg tokens - simple avg tokens`：

- 最大的是 `SFT-CoT`
- `length gap = 83.50`

这说明它最明显地把复杂题答得更长。

### DPO 的当前状态

`DPO-Adaptive` 这次训练虽然训练日志看起来正常，但在最终测试集上没有带来收益，反而显著退化：

- `overall accuracy = 0.1019`

更重要的是，它没有真正学成“简单题短答、复杂题长答”，而更像是“整体又变得偏长”。

所以如果后续还要继续做 DPO，建议优先检查：

- prompt 设计
- chosen / rejected 的格式一致性
- 是否显式告诉模型 `difficulty`
- simple 样本是不是还保留了太多解释性文字

## 13. 如果后续要继续实验，建议怎么做

建议优先顺序如下：

1. 先保留当前 `SFT-NoCoT` 和 `SFT-CoT` 结果作为稳定 baseline
2. 重做一版更强约束的 `DPO-Adaptive`
3. 在 DPO prompt 中显式加入 `difficulty`
4. 给 simple / complex 设计更清晰的一致输出规范
5. 重新做统一评测，对比：
   - `Base direct`
   - `Base cot`
   - `SFT-CoT`
   - `SFT-NoCoT`
   - `DPO-Adaptive v2`
